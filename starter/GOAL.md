# Graph Climbing Worker Goal

Give this same goal to every long-running worker. Do not insert a lane ID, claim ID, review round, or current task. Those values must be reconstructed from durable authorities.

```text
Operate as one persistent Graph Climbing worker for this repository. Repeatedly acquire, complete, verify, and reconcile graph-derived work until the authoritative product graph is complete or a defined stop condition is reached.

This goal is an execution protocol, not product truth and not a lease for one named task. Product meaning and official progress belong to the repository's existing ISA, specification, or equivalent product authority. Ownership, dependencies, gates, steering, and lease state belong to the repository's one operational ledger when present. Git, tests, reviews, runtime observations, dependency pins, and release artifacts are snapshot-bound evidence. Never preserve required state only in this goal or chat.

At start, resume, and after every reconciliation:

1. Read repository instructions and reconstruct the authority map, current checkout and base, product claims, execution graph, ledger state, evidence, and external gates. Resolve a stable owner identity, unique runtime/session incarnation, and exclusive mutation workspace. Never infer current work from stale chat.
2. Before steering or selection, resolve the execution profile and existing work. Before a ledger lease becomes ready, its central release owner immutably binds the lease ID to one stable vertical ID, canonical envelope hash, exclusive workspace, run epoch when applicable, and one pre-created authority-scope recovery barrier ID. The atomic claim binds the unique worker incarnation as the ledger assignee/actor; a stable human or team name is not an incarnation. Resume only when every binding matches and the recovery barrier is inactive; an active barrier prevents that incarnation from restarting or mutating. Intentional handoff requires proof the prior incarnation stopped and no reconciliation is in flight, then exclusive acquisition of its original workspace before rebinding the assignee. An unavailable workspace, missing or uncertain state, or any workspace transfer invokes recovery. Recovery starts only after atomically claiming the exact recovery barrier using the unique incarnation; a lost barrier claim stops. Its owner freezes new claims, proves every scope writer stopped, then replays or durably dispositions each incomplete reconciliation record while its exact lease remains active. A successful replay closes that lease without replacement. Only when no eligible record remains, resolve and durably import or disposition all Git, filesystem, runtime, and evidence state; if the vertical remains open, tombstone the old lease and create a new pre-bound lease plus successor recovery barrier. An N-worker replacement resumes only through the epoch barrier in step 6. Close the recovery barrier last with evidence. The barrier is non-stealable: owner loss keeps the scope frozen and requires external principal-authorized recovery that proves the owner stopped and performs a supported authoritative reset or full epoch replacement. Never infer failure or takeover authority from time alone. Multiple, mismatched, pre-epoch, or duplicate-incarnation leases fail closed. A ledgerless single writer reconstructs from product authority, Git, and evidence after context loss.
3. With an owned lease, ingest only steering explicitly targeting that lease at a declared safe boundary. Atomically claim the steering record, persist its applied effect or durable disposition, then close it last with evidence. If it changes base, scope, authority, run epoch, or topology, persist and link the replacement regraph/barrier record before closing; a failed handoff leaves steering unclosed. Without an owned lease, use the same claim–persist–close sequence for ready steering before deriving work.
4. Derive the current claim frontier. If an Execution Graph exists, derive its executable vertical frontier from verified predecessors, gates, cohort barriers, ownership, and file, read, runtime, authority, and checkout scopes.
5. For N mutating workers, claim only from one immutable run epoch published by its declared central owner before those workers start. It binds the product/graph base, execution-graph revision, an admitted candidate pool including future-ready verticals, canonical release-envelope hashes, and validation evidence for every pair whose scope-reservation lifetimes can overlap under any valid schedule. Current graph state still decides which admitted candidates are ready. A reservation lasts from claim through direct reconciliation or, for a cohort lane, through companion-join reconciliation. Incompatible work must be serialized by a real dependency or gate. Workers never publish or replace the epoch.
6. Initial epoch activation and replacement use the same coordinated stop-the-world regraph: durably pause or terminate every mutating worker; reconcile or explicitly withdraw all pre-epoch, claimed, active, or sealed lanes and companion joins; prove no mutation lease or reservation survives; publish the epoch; then resume the unchanged worker goals. A topology, base, scope, authority, candidate-set, or envelope change requires this barrier. If it cannot be proven, keep the old graph or stop.
7. Use ledger priority and stable ID as the deterministic candidate order. Verify the immutable pre-bound envelope hash, workspace, applicable epoch, and inactive recovery barrier, then atomically claim one exact candidate using the unique incarnation as ledger actor/assignee. Immediately re-read every binding and the barrier. An active barrier permits no mutation. Do not assume claim-time metadata flags share the claim compare-and-swap unless the installed ledger proves it. In an N-worker run, a lost race refreshes the frontier; in a serial ledger run, it disproves single-writer status. On mismatch, do not mutate; use authorized handoff or recovery. Never hold two mutating leases, invent custom locks, or continue from a cached frontier.
8. Without a ledger, proceed only after proving this is a single-mutating-worker run. Select one bounded claim-first vertical locally from the claim frontier; this runtime-local selection is the lease. If single-writer status is unknown, stop. Never use this profile for N workers.
9. Read the vertical's complete release envelope: stable claim closure, exact base, dependencies, write/read/runtime/authority scopes, checkout policy, first falsifying probe, stop rule, evidence destination, review gates, reconciliation owner, and companion join when applicable. In a claim-first single-writer run, report this envelope before mutation. Missing material fields block execution; do not guess them.
10. Implement the smallest complete slice inside that envelope. Run the first falsifying probe early, then focused adversarial tests, the full applicable gate, and required review against exact snapshots. Correct accepted findings and repeat only the affected verification loop. Do not harden governance, hooks, review control, or unrelated infrastructure unless the product vertical explicitly requires it.
11. Persist evidence, review dispositions, decisions, invalidations, and dependency pins in their owning surfaces. A parallel lane may seal as an evidence candidate but may not verify its covered claims or unlock ordinary successors before its companion join. Once the seal record is durable, release that worker's mutating lease while the cohort/epoch retains the lane's scope reservation through the join. The worker re-derives and may claim another ready, pre-admitted vertical only when it is compatible with every pending reservation. A straggler blocks its join and dependents, not independent admitted frontiers.
12. Reconcile across product authority, evidence, Git, and the ledger when present as one record-replayable logical transaction, not a claimed cross-store atomic write. Only the declared reconciliation owner in its exclusive authority workspace may enter the commit-to-close critical section; lease recovery or reassignment for that authority scope is forbidden until it exits. Persist a record binding `reconciliation_id`, vertical ID, input/output snapshots, and, when ledger-backed, exact lease ID, envelope hash, and run epoch when present. Verify all bindings and, when ledger-backed, that the lease remains active/current. Idempotently commit product truth and evidence, then close that exact ledger ID last with the reconciliation ID and commit reference when one exists. On failure, leave the record incomplete; ledger-backed runs invoke the stopped-writer recovery barrier, while a ledgerless single writer reconstructs from product truth, Git, evidence, and the record. Replay eligible incomplete records idempotently; tombstoned lease records are superseded without writing product truth. Downstream work stays blocked until all owning surfaces agree.
13. If closure makes a companion join ready, claim that exact join only when this worker satisfies its integration and checkout authority. The join runs combined probes and review through the same reconciliation transaction.
14. Release the completed lease, reconstruct durable state, derive the new frontier, and repeat. The graph and atomic claim results decide what this worker receives next; the worker does not carry forward yesterday's selection.

For a Beads ledger, the release owner persists immutable lease bindings before Ready. Each session uses a unique actor such as `<owner>@<incarnation>` and claims with `bd update <id> --claim --actor <unique-incarnation> --json`; Beads atomically stores that actor as assignee with `in_progress`. Re-read all bindings. Close the same exact ID last with `bd close <id> --reason "reconciliation_id=<reconciliation_id> commit=<commit_ref>" --suggest-next --json`, so the supported close reason persists both links. Use the same claim command for `recovery_via`; close that barrier last with a reason naming the replay, disposition, or replacement. Do not treat `--claim` plus metadata flags as one atomic binding operation unless the installed backend proves it. A failed claim is contention, not permission to proceed.

The same goal must be safe for N workers. Parallel mutation requires an operational ledger, explicit owners, independently falsifiable verticals, a validated admission frontier, and a pre-created companion join for every released cohort. Each worker holds at most one mutating lease; read-only verifier fan-out must target one immutable snapshot.

Mark this goal complete only when the product authority says all in-scope claims and required joins are verified, or an explicit authorized termination changes the desired state. No ready compatible work means an idle or blocked frontier, not proof of completion. Stop and report exact evidence for missing authority, ambiguous or multiple product authorities, invalid graph or ledger state, unsafe workspace ownership, scope collision, base drift, unresolved normative semantics, external approval, public write, live spend, or irreversible action. Never claim completion from confidence, elapsed time, a clean review alone, or context exhaustion.
```

## Required lane release envelope

The generic goal can remain task-free only when every executable vertical durably supplies:

- stable claim closure, vertical ID, dependencies, and a single-assignment ledger lease ID when used;
- exact release base and checkout policy;
- stable owner or eligible class; immutable workspace identity; unique incarnation bound atomically as ledger actor/assignee;
- write, read, runtime, and authority scopes;
- first falsifying probe and stop rule;
- evidence destination and required gates;
- product-authority write permission: `none`, named claim IDs, or `central`;
- canonical envelope hash for every ledger lease; plus immutable run-epoch ID, central owner, and validation evidence when N workers may claim concurrently;
- `recovery_via`: the exact pre-created, non-stealable authority-scope recovery barrier ID for every ledger-backed lease;
- `reconcile_via` for a released cohort lane, plus a companion whose `join_for` names the exact cohort.

If the ledger cannot represent these fields directly, link one durable release record from the vertical. Do not copy them into the goal.
